# 003 — Rate Limiting and Invalid-JSON Policy

## Status

Accepted

## Context

`docs/prompt.md` (original ToR v4.0, frozen since project start, superseded
in authority by ANCHOR_CORE.md/ANCHOR_PROTOCOL.md/ANCHOR_RULES.md per
ANCHOR_RULES.md Rule 2) specifies two connection-handling behaviours for
`onMessage` that were never carried forward into any ANCHOR document or
ROADMAP.md Epic:

```
onMessage: безопасный парсинг JSON (невалидный — закрыть соединение),
switch/case по action, проверка userId для всех кейсов кроме register,
login, reconnect.

Rate Limiting: более 15 пакетов/сек от одного соединения — немедленный
разрыв.
```

This created two problems, both flagged in `IMPLEMENTATION_STATUS.md`
KNOWN GAPS during the pre-Phase-10 audit:

1. **Rate limiting** has no home in ANCHOR_CORE.md/ANCHOR_PROTOCOL.md/
   ANCHOR_RULES.md at all — an undocumented requirement that could be
   lost permanently if not formalized before EPIC-10.1.
2. **Invalid JSON handling contradicts itself across documents.**
   `prompt.md` says "close the connection". `ANCHOR_PROTOCOL.md` already
   declares `error.invalid_json` as a valid Error Packet code — a code
   only makes sense if the client is expected to receive and parse it,
   which requires the connection to remain open. Separately, the
   already-implemented `error.server_full` (see `LobbyService::
   handleCreateRoom()`/`handleJoinRoom()`, Phase 2) resolves the
   analogous "reject and inform" case via a structured error packet, NOT
   via a raw connection close with a custom WS close code — establishing
   a real precedent already shipped and tested (90/90 lobby integration
   tests) that the project's actual trajectory favours informative error
   packets over abrupt closes wherever the protocol already defines a
   code for the situation.

EPIC-10.0 (`server.php`) already provisionally implemented "send
`error.invalid_json`, keep connection open" for malformed JSON and for
unknown/not-yet-wired actions, explicitly marked as provisional pending
this decision.

## Decision

**1. Invalid JSON / malformed packets → send `error.invalid_json`, do
NOT close the connection.**

This confirms and finalizes what EPIC-10.0 already implements
provisionally. Rationale: the error code's existence in
ANCHOR_PROTOCOL.md presupposes the client receives it; the
`error.server_full` precedent already establishes "inform via error
packet, don't close" as the project's real convention for rejectable
conditions. Abrupt closes are reserved for cases with no meaningful
application-level response (see Decision 2) or genuine abuse.

**2. Rate limiting: more than 15 packets/sec from a single connection →
immediately close the connection.**

Implemented exactly as specified in `prompt.md` — this directive does
not conflict with anything else and has genuine abuse-prevention value
(ANCHOR_RULES.md Rule 28 § VPS Awareness: 1 CPU/500MB RAM, a flood of
malformed-JSON packets must not be free to send indefinitely just
because each individual packet only receives a soft error response).

Combined, the two decisions are complementary rather than contradictory:
rate limiting is the actual abuse-prevention layer (applies uniformly to
ALL incoming messages regardless of validity), while `error.invalid_json`
remains a purely informational response for legitimate clients that
occasionally send a malformed packet.

**Mechanism** (new Connection Runtime Fields, ANCHOR_CORE.md Part 1 —
amended by this ADR):
```php
$connection->packetCount;      // int, счётчик пакетов в текущем окне
$connection->packetWindowStart; // int (unix timestamp), начало окна
```
Window: 1 second, reset on rollover. Threshold: `> 15` packets within the
window → `$connection->close()`, no error packet sent (the client is
being disconnected for abuse, not informed of a parseable mistake).

## Consequences

Positive:
- Both KNOWN GAPS from the pre-Phase-10 audit are closed.
- `server.php`'s provisional EPIC-10.0 behaviour is confirmed correct,
  no rework needed there beyond adding the rate-limit counter.
- Consistent with the already-shipped `error.server_full` precedent —
  no new architectural pattern introduced.

Negative:
- Two new Connection Runtime Fields added (ANCHOR_CORE.md Part 1 §
  Connection Runtime Fields updated — see amendment below).

## Compatibility

No protocol packet renamed, removed, or changed in meaning.
`error.invalid_json` semantics clarified (not changed — no prior
ANCHOR document ever stated it should close the connection).
No economy, database schema, or timer changes.

## Amendments to other ANCHOR documents

- `ANCHOR_CORE.md` Part 1 § Connection Runtime Fields: add
  `packetCount`, `packetWindowStart` to the allowed field list.
- `ANCHOR_PROTOCOL.md` § Error Packet: add an explicit note that
  `error.invalid_json` does not close the connection.
