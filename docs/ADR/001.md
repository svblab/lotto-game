# 001. Single Active Session Per User

## Status

proposed

## Context

The Lotto Game server uses Workerman and maintains runtime player state in memory. According to the architecture defined in ANCHOR_CORE.md, authenticated users are tracked through a centralized connection registry:

```php
$worker->userConnections[$userId] = $connection;
```

The registry serves multiple purposes:

* fast user lookup;
* reconnect support;
* double-login protection;
* player-to-connection mapping during gameplay.

The project is intentionally designed for simplicity, predictability, and low operational complexity. According to ANCHOR_RULES.md:

* hidden refactoring is prohibited;
* protocol changes are prohibited;
* Player and Room structures must remain unchanged;
* server-side behavior must remain deterministic.

Allowing multiple simultaneous authenticated sessions for a single account would introduce ambiguity regarding:

* ownership of game actions;
* validity of session tokens;
* reconnect behavior;
* handling of duplicate commands from multiple devices.

Such behavior would significantly increase implementation complexity and create additional synchronization risks.

## Decision

The server shall enforce a strict "single active session per user" policy.

Rules:

1. A user account may have only one active authenticated connection at any time.

2. During login, the server must verify whether the user already exists in the runtime connection registry.

3. If an active authenticated session already exists for the same user, the login request must be rejected.

4. The server shall return an authentication error:

```text
User already logged in
```

5. Temporary connection loss must be handled exclusively through the reconnect mechanism.

6. Reconnect is the only supported method for restoring access to an existing game session.

7. Server restarts invalidate all active sessions and reconnect tokens.

8. Logout functionality may remove the user from the connection registry in the future, but logout implementation is outside the scope of this decision.

## Consequences

### Positive

* Eliminates double-login conflicts.
* Prevents multiple devices from issuing game commands simultaneously.
* Simplifies reconnect logic.
* Simplifies session ownership rules.
* Keeps gameplay state deterministic.
* Reduces memory and synchronization overhead.
* Aligns with the project's lightweight Workerman architecture.

### Negative

* Users cannot remain logged in from multiple devices simultaneously.
* Opening a second browser, tab, or device requires termination of the existing session first.
* Future support for multi-device sessions would require a separate architectural decision.

### Follow-Up Actions

* Implement login-time active session validation.
* Maintain the user connection registry as the authoritative source of authenticated sessions.
* Integrate reconnect logic with the single-session policy.
* Add manual tests covering duplicate login attempts and reconnect behavior.
