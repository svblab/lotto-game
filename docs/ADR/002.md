# 002 — Extract GameFinishService

Status: Accepted

Date: 2026-07-02

## Context

GameService has grown to include multiple independent responsibilities:

- game initialization;
- turn processing;
- coordination of VictoryService and ApartmentService;
- complete game finalization.

The last responsibility contains its own database transaction,
statistics generation, protocol broadcasting and room destruction,
making it a separate lifecycle concern.

According to ANCHOR_CORE.md File Size Policy and the architectural
module boundaries, game finalization should be isolated into its own
service before implementing Phase 8 (Reconnect & AFK).

## Decision

Create a new service:

src/Game/GameFinishService.php

The existing finishGame() implementation will be moved from
GameService into GameFinishService without changing its public
behaviour.

GameService will keep the orchestration role and delegate game
finalization to GameFinishService.

No protocol packets, economy rules, timers, room structures or
player structures are modified.

## Consequences

Positive:

- reduces GameService size;
- improves separation of responsibilities;
- keeps VictoryService and ApartmentService unchanged;
- prepares architecture for Phase 8;
- minimizes future merge conflicts.

Negative:

- one additional service dependency is introduced.

## Compatibility

This ADR introduces no protocol changes.

This ADR introduces no economy changes.

This ADR introduces no database schema changes.

Existing integration tests must continue to pass without
modification.
