# 005 — Connection-Level Global Capacity Gate (WS Close Code 4001)

## Status

Accepted

## Context

`docs/prompt.md` (line 41, original spec, never previously formalized in
an ANCHOR document — same class of gap already closed once for rate
limiting and invalid-JSON policy in ADR-003) states:

> Глобальный лимит игроков: при превышении 150 — закрыть соединение с
> кодом 4001 и error.server_full.

This describes a check distinct from the existing global-capacity check
in `LobbyService::handleCreateRoom()`/`handleJoinRoom()` (FIX-7 /
ADR-004): that check runs at room-join time, against
`RoomManager::getTotalPlayerCount()` — the count of players actually
seated in rooms, an already-authenticated subset. `prompt.md` describes
a check at the connection layer itself, before authentication, against
all live sockets — a resource-protection gate for the Workerman worker
process, not a game-capacity rule.

`server.php` had no such gate. `onWebSocketConnected` unconditionally
initialized connection fields and sent `hello` for every accepted TCP
connection, regardless of how many connections were already open.

Technical constraint investigated before implementation: the installed
Workerman version (`vendor/workerman/workerman`) has no built-in API to
close a WebSocket connection with an explicit close-frame status code.
`TcpConnection::close($data, $raw)` sends `$data` verbatim (bypassing
`Protocols\Websocket::encode()` when `$raw = true`) and then tears down
the socket — it does not construct a close frame itself. A correct close
frame (RFC 6455 §5.5.1) has to be built by hand: opcode `0x8`
(`"\x88"` as the first byte, FIN=1, no extension bits), a single-byte
length (server frames are never masked, and the payload here is always
short), followed by a 2-byte big-endian status code and an optional
UTF-8 reason.

## Decision

1. **New reusable helper `closeWithCode(object $connection, int $code,
   string $reason = '')`** in `src/Core/Helpers.php`, alongside
   `sendJson`/`sendError`. Builds the close frame by hand as described
   above and sends it via `$connection->close($frame, true)`. Payload is
   capped at 125 bytes (RFC 6455 §5.2 single-byte length field limit) —
   defensive only, never expected to bind in practice given how short
   `$reason` is here.

2. **Gate added at the very top of `onWebSocketConnected` in
   `server.php`**, before any connection-field initialization and before
   `hello` is sent:
   ```php
   if (count($worker->connections) > Constants::MAX_TOTAL_PLAYERS) {
       sendError($connection, 'error.server_full', 'Server is full');
       closeWithCode($connection, 4001, 'server_full');
       return;
   }
   ```
   Both the JSON `error.server_full` packet (normal protocol-encoded
   text frame, via the existing `sendError()` path) and the raw close
   frame with code 4001 are sent, in that order — belt-and-suspenders:
   a client that only inspects `WebSocket.onclose.code` still learns
   why, and a client that reads the message body before the close event
   fires also gets the structured error.

3. **Comparison operator is `>`, not `>=`** (unlike the `>=` used in
   `LobbyService`'s room-player checks). This is deliberate, not an
   inconsistency: Workerman registers a new connection into
   `$worker->connections` at TCP-accept time
   (`Worker::acceptTcpConnection()`), before the WebSocket handshake
   completes and before `onWebSocketConnected` fires — so by the time
   this gate runs, `count($worker->connections)` already includes the
   connection being evaluated. Using `>` therefore yields the same
   effective capacity as `LobbyService`'s pre-increment `>=` checks:
   exactly `Constants::MAX_TOTAL_PLAYERS` concurrent connections are
   allowed; the `(N+1)`-th is rejected.

4. **New WS close code registry** added to `ANCHOR_PROTOCOL.md` (§
   WebSocket Close Codes) — code `4001` documented as the sole entry for
   now, in the RFC 6455 private-use range (4000-4999). Table format
   established so future application-specific close codes have a place
   to be documented (an ADR is required to add a new row).

## Consequences

Positive:
- Formalizes an until-now-undocumented `prompt.md` requirement, closing
  another prompt.md-vs-ANCHOR gap (same pattern as ADR-003).
- Protects the Workerman worker process itself from unbounded connection
  growth, independent of and prior to any room/game-level capacity rule.
- `closeWithCode()` is a general-purpose helper — any future
  application-specific close code can reuse it without re-deriving the
  frame format.

Negative / follow-up:
- This is a single-process, in-memory check (`$worker->connections`) —
  it protects one Workerman worker instance. `$worker->count = 1` per
  `server.php`'s bootstrap, so this is a non-issue today; would need
  revisiting if the project ever moves to multiple worker processes,
  since the count would then need to be shared across workers rather
  than read from local memory.
- No client-facing i18n for close-code reasons is planned scope-wise
  here (matches the existing note in ADR-004 for `error.room_full`) —
  the JSON `error.server_full` packet is the primary UX channel; the
  close code is a secondary/redundant signal for clients that only
  observe `onclose`.

## Compatibility

New WS close code, new helper function — no existing packet, field, or
close behavior renamed or removed. `error.server_full`'s existing
semantics (ADR-004: reserved exclusively for the global limit) are
unchanged; this ADR is the connection-layer counterpart, not a
redefinition.

## Amendments to other ANCHOR documents

- `ANCHOR_PROTOCOL.md`: new § WebSocket Close Codes, code `4001`
  documented.
