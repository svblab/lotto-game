# 006 — Generic `auth_required` Router Guard

## Status

Accepted

## Context

`docs/prompt.md` (Фаза 1, `onMessage` bullet) specifies:

> onMessage: безопасный парсинг JSON (невалидный — закрыть соединение),
> switch/case по action, проверка userId для всех кейсов кроме register,
> login, reconnect.

This is the second half of EPIC-10.2 (Protocol error handling); the first
half (connection-level `MAX_TOTAL_PLAYERS` gate) was formalized and
implemented separately in ADR-005. This ADR covers the remaining,
previously-deferred item: a single, generic authentication check applied
by the router itself, before any action-specific handler runs, rather
than duplicated inside every future handler.

`error.auth_required` already existed in `ANCHOR_PROTOCOL.md`'s error
code registry, but — like `error.server_full`/`error.room_full` before
FIX-7/ADR-004, and the connection-level gate before ADR-005 — it had no
documented semantics: no note said which actions were exempt or when
exactly it fires. Nothing in `server.php` sent it at all.

Note: `prompt.md`'s "невалидный — закрыть соединение" (close on invalid
JSON) for this same bullet was already superseded by ADR-003
(`error.invalid_json` does not close the connection) — not revisited
here, only the auth-check portion of this bullet is in scope.

## Decision

1. **Guard placed in `onMessage`**, immediately after the existing
   "missing/invalid action field" check and before the (still empty)
   action dispatcher:
   ```php
   $authExemptActions = ['register', 'login', 'reconnect'];
   if ($connection->userId === null && !in_array($action, $authExemptActions, true)) {
       sendError($connection, 'error.auth_required', 'Authentication required');
       return;
   }
   ```
2. **Exempt set is exactly `{register, login, reconnect}`**, matching
   `prompt.md` verbatim. `ping` is not listed in this set because it is
   already handled earlier in `onMessage` (returns before this point is
   reached) — including it here would be redundant, not a behavioral
   gap.
3. **Checked once, generically, by the router — not per-handler.** This
   is a protocol-routing rule (which actions require an established
   session), not business logic belonging to any single future handler,
   so it belongs in `server.php` under Rule 15 (dependency
   wiring/routing permitted here).
4. `ANCHOR_PROTOCOL.md` § Error Packet amended with a note for
   `error.auth_required`, mirroring the existing notes for
   `error.invalid_json` (ADR-003) and `error.server_full`/`error.room_full`
   (ADR-004).

## Consequences

Positive:
- Closes another `prompt.md`-vs-ANCHOR documentation gap (same recurring
  pattern as ADR-003/004/005).
- Future handlers (EPIC-10.3-10.6) never need to re-implement this
  check; they can assume `$connection->userId` is set by the time they
  run, for any non-exempt action.

Side effect worth naming explicitly (not a defect, but a real behavior
change verified in `tests/Manual/test_server_bootstrap.php`): the
dispatcher's `default => error.invalid_json` fallback (still the only
case in the `match()`, since no real handlers are wired yet) is now
**unreachable for an unauthenticated connection sending any non-exempt
action** — the guard intercepts first and returns `error.auth_required`
instead. The fallback remains reachable only for the exempt actions
themselves (`register`/`login`/`reconnect`, not yet wired to real
handlers until EPIC-10.3) and, once authentication exists, for
authenticated connections sending a genuinely unknown action.

## Compatibility

No packet, field, or action renamed/removed. `error.auth_required` is an
existing code in the registry — this ADR only documents when it fires
and makes the router actually send it; no new code introduced.

- `ANCHOR_PROTOCOL.md` § Error Packet: note added for `error.auth_required`.
