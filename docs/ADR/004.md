# 004 — Distinct `error.room_full` Code and Join-Order Fix

## Status

Accepted

## Context

`LobbyService::handleJoinRoom()` used `error.server_full` for two
semantically different conditions:

1. The genuine global limit — `getTotalPlayerCount($worker) >=
   Constants::MAX_TOTAL_PLAYERS` (whole server at capacity).
2. A single, otherwise-valid room having reached its own `max_players`
   (`count($room['players']) >= $room['max_players']`) — a per-room
   condition with no relation to server-wide capacity.

`ANCHOR_PROTOCOL.md`'s error registry had no dedicated code for the
per-room case, so both conditions were reported to the client identically
as `error.server_full`. A client cannot distinguish "the whole server is
full, nothing to do" from "this one room is full, try another or create
one" — the two call for different UI/UX responses.

Additionally, the two checks were ordered room-capacity-first,
server-capacity-second. Under this order, if both conditions were true
simultaneously (room full AND server full), the client would receive the
less severe/less accurate `error.server_full` used as "room is full"
label rather than the true, more actionable global-capacity error —
harmless today given both used the same code, but became a real ordering
bug once the codes diverge (Decision 1 below).

User confirmed both problems explicitly: the codes must differ, and
server-wide capacity must be checked before room capacity.

## Decision

1. **New protocol error code `error.room_full`** (`ANCHOR_PROTOCOL.md` §
   Error Packet — addition, not a rename; permitted by the Protocol
   Compatibility Rule without a version bump). Used exclusively for a
   specific room being at its own `max_players`. `error.server_full`
   remains reserved exclusively for the global `MAX_TOTAL_PLAYERS` limit.

2. **Check order in `handleJoinRoom()` fixed**: the global server-capacity
   check now runs before the per-room capacity check. If both are true,
   the client receives `error.server_full` (the more actionable,
   higher-severity condition), never `error.room_full`.

`handleCreateRoom()` required no change — it only ever performed the
global check; it never had a room-capacity check to conflict with (a
room being created cannot itself be full).

## Consequences

Positive:
- Client can distinguish "server full" from "this room full" and react
  accordingly (retry a different room vs. no viable action).
- Check order is now deterministic and matches severity: the more
  general condition (server full) is reported first when both apply.

Negative:
- One new protocol error code to document/localize on the client
  (Phase 11, i18n `error.*` key translation — not yet implemented on the
  client side; noted for Phase 11 scope, no action required now).

## Compatibility

`error.room_full` is a new code, no existing code renamed, removed, or
had its meaning changed. `error.server_full` semantics are unchanged,
only clarified as exclusively global-scope.
No economy, database schema, or timer changes.

## Amendments to other ANCHOR documents

- `ANCHOR_PROTOCOL.md` § Error Packet: `error.room_full` added to the
  code registry; note added distinguishing it from `error.server_full`
  and documenting join-order precedence.
